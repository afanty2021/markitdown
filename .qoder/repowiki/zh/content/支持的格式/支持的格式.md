## 简介
markitdown 是一个简单的基于文本的文档阅读器，专为大语言模型（LLM）使用而设计。它能够将常见的文件类型或网页转换为 Markdown 格式。本文档系统性地列出了 markitdown 支持的所有文件格式，包括文档、电子表格、演示文稿、多媒体、压缩文件、网页内容以及其他特殊格式。对于每种格式，文档说明了其对应的转换器实现、转换原理、预期输出示例以及潜在限制，并解释了文件类型识别机制如何与特定转换器关联。

## 文档格式

### PDF
PDF 文件由 `_pdf_converter.py` 转换器处理。该转换器利用 `magika` 库进行文件类型识别，并通过 `pdfplumber` 或 `PyPDF2` 等库提取文本内容。转换原理是解析 PDF 的页面结构，提取其中的文本流，并将其格式化为 Markdown。

**预期输出示例**:
```markdown
# 文档标题

这是 PDF 文档中的第一段文字。

## 章节标题

- 列表项一
- 列表项二
```

**潜在限制**:
- 复杂的布局和表格可能无法完美保留。
- 扫描的 PDF（图像）需要 OCR 支持，否则无法提取文本。

**Section sources**
- [_pdf_converter.py](file://packages/markitdown/src/markitdown/converters/_pdf_converter.py)

### DOCX
DOCX 文件由 `_docx_converter.py` 转换器处理。该转换器首先使用 `pre_process_docx` 函数预处理文档，特别是将 Office Math Markup Language (OMML) 数学公式转换为 LaTeX 格式。然后，它使用 `python-docx` 库解析文档结构，提取段落、列表、表格和图像，并将其转换为 Markdown。

**转换原理**:
1.  **预处理**: 解压 DOCX 文件（ZIP 格式），在内存中处理 `word/document.xml` 等文件，将 OMML 公式转换为 LaTeX。
2.  **解析**: 使用 `python-docx` 读取预处理后的文档。
3.  **转换**: 遍历文档元素，将每个元素映射为相应的 Markdown 语法。

**预期输出示例**:
```markdown
# 文档标题

这是一个包含数学公式的段落：$E = mc^2$。

## 章节

- 项目符号列表
1.  编号列表
```

**潜在限制**:
- 高度复杂的样式或嵌套表格可能无法完全准确转换。
- 某些特殊字体或图形可能丢失。

### HTML
HTML 文件由 `_html_converter.py` 转换器处理。该转换器使用 `_markdownify.py` 中定义的 `_CustomMarkdownify` 类，这是一个基于 `markdownify` 库的自定义转换器。它将 HTML 标签系统地转换为等效的 Markdown 语法。

**转换原理**:
- 重写 `markdownify` 的默认行为，例如确保标题前有换行符。
- 移除 JavaScript 链接。
- 对 URI 进行正确转义以避免与 Markdown 语法冲突。
- 处理图片、链接、列表、表格等 HTML 元素。

**预期输出示例**:
```markdown
# 页面标题

这是一段来自 HTML 的文本。

[链接文本](https://example.com)

![图片描述](image.jpg)
```

**潜在限制**:
- 动态生成的内容（通过 JavaScript）不会被转换。
- 复杂的 CSS 样式无法在 Markdown 中体现。

## 电子表格

### XLSX 和 XLS
XLSX 和 XLS 文件分别由 `_xlsx_converter.py` 和 `_xls_converter.py` 转换器处理。这些转换器使用 `pandas` 或 `openpyxl`/`xlrd` 库读取电子表格数据。转换原理是将每个工作表（worksheet）转换为一个 Markdown 表格。

**预期输出示例**:
```markdown
| 列1 | 列2 | 列3 |
| --- | --- | --- |
| 数据1 | 数据2 | 数据3 |
| 数据4 | 数据5 | 数据6 |
```

**潜在限制**:
- 公式和宏不会被计算或转换，只会显示其结果值。
- 图表和图形会被忽略。
- 复杂的单元格合并可能无法完美呈现。

**Section sources**

## 演示文稿

### PPTX
PPTX 文件由 `_pptx_converter.py` 转换器处理。该转换器使用 `python-pptx` 库读取演示文稿。转换原理是遍历每一张幻灯片，提取幻灯片上的文本框内容，并将每张幻灯片的内容用分页符（`---`）分隔。

**预期输出示例**:
```markdown
# 幻灯片 1 标题

- 要点一
- 要点二

---

# 幻灯片 2 标题

这是第二张幻灯片的正文。
```

**潜在限制**:
- 幻灯片的视觉布局（如位置、动画）无法在 Markdown 中表示。
- 图像和图表会被忽略，仅保留文本。

**Section sources**

## 多媒体

### 图像
图像文件（如 JPG, PNG）由 `_image_converter.py` 转换器处理。该转换器的主要功能是提取图像的元数据（通过 `_exiftool.py`）和生成图像描述（通过 `_llm_caption.py`）。

**转换原理**:
1.  **元数据提取**: 调用 `exiftool` 命令行工具，通过 `subprocess` 获取图像的 EXIF 信息（如相机型号、拍摄时间）。
2.  **图像描述**: （可选）使用 LLM（如 GPT-4V）分析图像内容并生成文本描述。

**预期输出示例**:
```markdown
![图像文件名](data:image/jpeg;base64,/9j/4AAQSkZJR...)

**元数据**:
- 相机: Canon EOS 5D Mark IV
- 日期: 2023:10:27 14:30:25

**描述**: 这张照片显示了一座被雪覆盖的山峰，在日出时分显得格外壮丽。
```

**潜在限制**:
- 需要安装 `exiftool` 工具才能提取元数据。
- LLM 描述功能需要有效的 API 密钥和网络连接。

### 音频
音频文件（如 MP3, WAV）由 `_audio_converter.py` 转换器处理。该转换器依赖 `_transcribe_audio.py` 模块进行语音识别。

**转换原理**:
1.  **格式处理**: 对于 MP3 等非 WAV 格式，使用 `pydub` 库将其转换为 WAV 格式。
2.  **语音识别**: 使用 `speech_recognition` 库调用 Google Web Speech API 将音频中的语音转换为文本。

**预期输出示例**:
```markdown
这是从音频文件中转录出的语音内容。
```

**潜在限制**:
- 需要安装 `markitdown[audio-transcription]` 可选依赖。
- 转录质量高度依赖于音频的清晰度和背景噪音。
- 依赖 Google 的在线 API，需要网络连接。

## 压缩文件

### ZIP
ZIP 文件由 `_zip_converter.py` 转换器处理。该转换器本身不直接转换内容，而是作为一个容器处理器。它会解压 ZIP 文件，并尝试使用其他转换器处理其中的每一个文件。

**转换原理**:
- 使用 Python 的 `zipfile` 模块读取 ZIP 文件。
- 遍历压缩包内的每个文件。
- 对每个文件，递归调用主转换流程，根据文件类型选择合适的转换器。

**预期输出示例**:
输出将是 ZIP 包内所有文件转换结果的集合，每个文件的结果前可能会有文件名作为标题。

**潜在限制**:
- 无法处理加密的 ZIP 文件。
- 如果 ZIP 包内文件类型不支持，这些文件将被忽略。

**Section sources**
- [_zip_converter.py](file://packages/markitdown/src/markitdown/converters/_zip_converter.py)

## 文件类型识别机制
markitdown 使用一个多层次的文件类型识别机制来确定使用哪个转换器。该机制在 `_markitdown.py` 文件中的 `_get_stream_info_guesses` 方法中实现。

**识别流程**:
1.  **基础猜测 (Base Guess)**: 基于文件扩展名、MIME 类型、文件名或 URL 创建一个初始的 `StreamInfo` 对象。
2.  **MIME 类型/扩展名推断**: 如果只有扩展名，使用 `mimetypes` 库推断 MIME 类型；如果只有 MIME 类型，则推断可能的扩展名。
3.  **内容识别 (Magika)**: 使用 `magika` 库分析文件的二进制内容，以获得更准确的文件类型预测。`magika` 是一个基于机器学习的文件类型识别工具。
4.  **兼容性检查**: 将 `magika` 的猜测结果与基础猜测进行比较。如果它们兼容，则合并信息；如果不兼容，则保留两个独立的猜测。
5.  **转换器选择**: 系统会遍历所有注册的转换器（按优先级排序），并调用每个转换器的 `accepts` 方法。该方法会检查 `StreamInfo` 对象中的信息（如扩展名、MIME 类型）是否与该转换器支持的格式匹配。第一个返回 `True` 的转换器将被用于执行转换。

## 转换器注册与优先级
所有转换器都在 `MarkItDown` 类的 `enable_builtins` 方法中注册。转换器的执行顺序由其优先级（`priority`）决定。

**优先级规则**:
- 优先级数值越低，优先级越高，越早被尝试。
- **特定文件格式优先级 (PRIORITY_SPECIFIC_FILE_FORMAT = 0.0)**: 分配给 `.docx`, `.pdf`, `.xlsx` 等特定格式的转换器。
- **通用文件格式优先级 (PRIORITY_GENERIC_FILE_FORMAT = 10.0)**: 分配给 `PlainTextConverter`, `HtmlConverter`, `ZipConverter` 等更通用的转换器。

这种设计确保了对于一个 `.docx` 文件，系统会优先尝试 `DocxConverter`，只有在失败后才会尝试更通用的 `HtmlConverter` 或 `PlainTextConverter`。
